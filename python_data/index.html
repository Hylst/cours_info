üìñ Comprendre Numpy (Concepts Cl√©s)

Le Probl√®me : Les listes Python sont flexibles (on peut m√©langer types et tailles), mais cette flexibilit√© a un co√ªt : chaque √©l√©ment est un objet Python complet, dispers√© en m√©moire. Pour des millions de calculs, c'est beaucoup trop lent.

La Solution Numpy : Numpy introduit les ndarray (N-dimensional arrays), des tableaux typ√©s et homog√®nes. Tous les √©l√©ments ont le m√™me type (int32, float64, etc.) et sont stock√©s dans un bloc m√©moire contigu.

Contigu√Øt√© M√©moire : C'est le secret de la vitesse. Le CPU adore les donn√©es contigu√´s car il peut les charger par blocs dans le cache (principe de localit√©). Une boucle for Python est interpr√©t√©e ; une op√©ration Numpy est ex√©cut√©e en C/Fortran optimis√©.

Le Broadcasting : Permet d'appliquer des op√©rations entre tableaux de dimensions diff√©rentes sans boucle explicite. Numpy "√©tend" virtuellement le plus petit tableau pour correspondre au plus grand. C'est magique et ultra-performant.

Le Reshaping : Critique en Deep Learning. Une image 28x28 pixels devient un vecteur de 784 √©l√©ments pour un r√©seau de neurones (arr.reshape(-1)). Un batch de 100 images devient un tenseur de forme (100, 784).

Le Broadcasting √©vite les boucles `for` en propageant les dimensions plus petites sur les plus grandes.

üí° Contexte R√©el : Dans une application d'IA, on re√ßoit souvent des milliers d'images. On les "stack" dans un seul tenseur Numpy pour les envoyer d'un coup au GPU. C'est l'essence du Batch Processing.

üìñ Comprendre l'Acquisition (Concepts Cl√©s)

D√©finition : L'acquisition de donn√©es (Data Acquisition ou DAQ) va bien au-del√† de la simple collecte. C'est un processus continu o√π les donn√©es sont constamment collect√©es, transform√©es, valid√©es et valoris√©es pour l'analyse et la prise de d√©cision.

Le Principe GIGO : "Garbage In, Garbage Out". De mauvaises donn√©es en entr√©e produisent de mauvais r√©sultats. La qualit√© des donn√©es est fondamentale : inexactitudes, incoh√©rences ou donn√©es obsol√®tes ruinent les mod√®les.

Sources Internes : Donn√©es des syst√®mes d'entreprise (ERP, CRM), logs applicatifs, bases de donn√©es transactionnelles. Souvent en silos, n√©cessitent d√©silottage.

Sources Externes : APIs REST (donn√©es en temps r√©el), Open Data, Web Scraping, capteurs IoT. Attention √† la conformit√© (RGPD) et aux CGU.

Le Pipeline ETL/ELT : Extract (extraire des sources), Transform (nettoyer, normaliser, enrichir), Load (charger dans un Data Warehouse ou Data Lake). C'est l'√©tape cruciale de pr√©paration qui peut repr√©senter jusqu'√† 90% du temps d'un projet.

Cl√©s du Succ√®s : Pertinence m√©tier (collecter les bonnes donn√©es), Rigueur technique (les pr√©parer correctement), Vigilance √©thique/l√©gale (RGPD, consentements, anonymisation).

üí° Contexte R√©el : Dans un projet ML, la pr√©paration des donn√©es (nettoyage, gestion des valeurs manquantes) peut repr√©senter jusqu'√† 90% du temps du Data Scientist. C'est l'√©tape ETL/ELT.

Machine Learning

Scikit-Learn (Sklearn)

üìñ Comprendre le Machine Learning (Concepts Fondamentaux)

D√©finition : Le Machine Learning (ML) est une branche de l'IA o√π les algorithmes apprennent des patterns √† partir de donn√©es, sans √™tre explicitement programm√©s. Contrairement √† la programmation classique (r√®gles ‚Üí r√©sultat), en ML on fournit des donn√©es et des r√©sultats attendus, et l'algorithme d√©duit les r√®gles.

Supervis√© vs Non-Supervis√© : En apprentissage supervis√©, on a des labels (la "r√©ponse" attendue) pour entra√Æner le mod√®le. En non-supervis√©, on cherche des structures cach√©es dans les donn√©es sans labels (clustering, r√©duction de dimension).

Le Train/Test Split : On divise TOUJOURS les donn√©es en deux parties. Le mod√®le apprend sur le train set (80%) et est √©valu√© sur le test set (20%) qu'il n'a jamais vu. C'est crucial pour mesurer la g√©n√©ralisation.

Overfitting vs Underfitting : Un mod√®le overfit quand il apprend "par c≈ìur" le train set mais √©choue sur le test (trop complexe). Un mod√®le underfit quand il est trop simple pour capturer les patterns. L'objectif est le juste milieu.

Preprocessing : Les mod√®les ne comprennent que les nombres. Il faut encoder le texte (OneHotEncoder), scaler les valeurs num√©riques (StandardScaler), et g√©rer les valeurs manquantes.

Workflow Industrialis√© (Pipeline)
üìñ Concepts Avanc√©s (Overfitting, M√©triques, GridSearch)

Cross-Validation : Pour un score plus robuste, on d√©coupe les donn√©es en K parties (Folds). On entra√Æne K fois le mod√®le en utilisant √† tour de r√¥le chaque fold comme test set. Le score final est la moyenne.

GridSearchCV : Teste automatiquement toutes les combinaisons d'hyperparam√®tres (ex: nombre d'arbres, profondeur max) pour trouver la meilleure configuration. Attention : computationnellement co√ªteux !

Pipelines : Encapsulent preprocessing + mod√®le dans un seul objet. Avantage : √©vite le "data leakage" (fuite d'information du test set) et simplifie le d√©ploiement.

M√©trique Cas d'usage Formule Intuitive
Accuracy Classes √©quilibr√©es (Vrai Pos + Vrai Neg) / Total
Recall (Rappel) D√©tection fraude/maladie (Ne rien rater) Vrai Pos / (Vrai Pos + Faux Neg)
Precision Anti-Spam (Ne pas filtrer les bons emails) Vrai Pos / (Vrai Pos + Faux Pos)
F1-Score Compromis Pr√©cision/Rappel (G√©n√©raliste) Moyenne harmonique

Deep Learning

R√©seaux de Neurones & Keras

üìñ Comprendre le Deep Learning (Concepts Fondamentaux)

D√©finition : Le Deep Learning est une branche du Machine Learning utilisant des r√©seaux de neurones artificiels √† plusieurs couches (d'o√π "deep"). Contrairement au ML classique o√π on con√ßoit manuellement les features, le DL apprend automatiquement des repr√©sentations de plus en plus abstraites.

Le Neurone Artificiel : C'est l'unit√© de base. Il re√ßoit des entr√©es (x‚ÇÅ, x‚ÇÇ, ...), les multiplie par des poids (w‚ÇÅ, w‚ÇÇ, ...), fait la somme, ajoute un biais, puis applique une fonction d'activation (non-lin√©arit√©). Formule : output = activation(Œ£(wi * xi) + b)

Backpropagation : L'algorithme d'apprentissage. On calcule l'erreur (Loss) entre la pr√©diction et la v√©rit√©, puis on "propage" cette erreur en arri√®re dans le r√©seau pour ajuster les poids. C'est la descente de gradient appliqu√©e couche par couche.

Pourquoi "Deep" ? Plus il y a de couches, plus le r√©seau peut apprendre des abstractions complexes. La 1√®re couche d√©tecte des bords, la 2√®me des formes, la 3√®me des objets, etc. C'est ce qui rend le DL si puissant pour les images, le texte et l'audio.

GPU / TPU : L'entra√Ænement DL n√©cessite des millions de calculs matriciels. Les GPU (cartes graphiques) sont optimis√©s pour ce type de calcul parall√®le. Sans GPU, l'entra√Ænement peut prendre des jours au lieu d'heures.

Transfer Learning : Utiliser un mod√®le pr√©-entra√Æn√© (ex: ResNet, BERT) et ne r√©-entra√Æner que la derni√®re couche. Gain de temps et de donn√©es colossal.
PyTorch vs Keras : Keras est id√©al pour le prototypage rapide. PyTorch est privil√©gi√© par la recherche pour sa flexibilit√© et son mode imp√©ratif.

Cheat Sheet Architecture DL

D√©finition : Le MLOps (Machine Learning Operations) est l'ensemble des pratiques pour d√©ployer et maintenir des mod√®les ML en production de mani√®re fiable. C'est le DevOps appliqu√© au Machine Learning.

Le Probl√®me : Un mod√®le qui reste dans un Notebook n'a aucune valeur business. 87% des projets ML ne d√©passent jamais le stade du prototype. Le MLOps vise √† r√©duire ce "gap" entre exp√©rimentation et production.
